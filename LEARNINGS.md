This document summarizes the key takeaways, observations, and implementation learnings gathered while working on:

- Issue: @addyosmani/AudioMassAI/issues/3
- PR: @addyosmani/AudioMassAI/pull/5

1. Summarization model evaluation

1.1 bart-large-cnn memory behavior

- Loading bart-large-cnn causes the system to become unresponsive during the initial model load.

- Once the model is loaded, subsequent summarizations no longer cause unresponsiveness.

- The initial loading phase is the primary bottleneck and can lock a 32GB RAM desktop for several minutes.

- Because of this, bart-large-cnn is not viable for production or for environments with limited resources (mobile support was explicitly a concern).

1.2 distilbart-cnn-6-6 fallback exploration

- As part of evaluating (fallback to a smaller model), distilbart-cnn-6-6 was tested.

- On desktop testing, distilbart-cnn-6-6 does not make the device unresponsive.

- It appears significantly less memory-intensive than bart-large-cnn during loading.

- No quantitative memory measurements were taken, but qualitatively:
  * No freezes
  * No UI lockups
  * Summarization steps ran smoothly

This makes distilbart-cnn-6-6 a more realistic fallback option compared to bart-large-cnn.

2. Chrome Summarizer API integration

This was documented for issue reporting and reproducibility.

3. Summarization quality comparison

3.1 Browser-native Summarizer API

- The built-in Chrome Summarizer API produces good-quality summaries in general.

3.2 Xenova/t5-small

- Summaries produced using Xenova/t5-small were not coherent and appeared more like “random sentences” rather than meaningful summaries.

3.3 distilbart-cnn-6-6

- Summaries generated by distilbart-cnn-6-6 did not significantly improve quality compared to t5-small for the test input.

This reinforced the case for relying on the Chrome Summarizer API where available.

4. Model loading UX improvements

From the PR work:

- Improved the UI to indicate which API/model is being used.

- Added “Ship with distilbart for browsers without the native API” handling.

- Updated the confirmation modal to reflect the actual size of the model (`~284MB`).

- These changes were reviewed and positively acknowledged.

5. Takeaways for fallback strategy

Based on testing and discussion:

- bart-large-cnn is unsuitable due to severe memory load.

- distilbart-cnn-6-6 is a viable fallback candidate but still heavy.

- A fallback may be removed entirely if reliability becomes the priority.

- A progressive, platform-aware strategy is preferred:
  * Use Chrome Summarizer API where available.
  * Potentially avoid fallback on mobile.
  * Possibly avoid fallback entirely for first release.

6. Summary of key decisions

- High-memory models are not feasible for broad user bases.

- Chrome’s native summarizer currently delivers the best quality and UX.

- A lightweight fallback must be carefully evaluated for memory impact.
